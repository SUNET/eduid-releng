# SLSA Gap Assessment and Implementation Plan

This note captures the SLSA-related analysis performed for the eduID release-engineering repository.

It is intentionally repo-specific. The goal is not to restate generic supply-chain guidance, but to describe how SLSA concepts apply to this repository's current build and promotion model.

## Scope

This repository is the release orchestration layer for eduID. It:

- selects upstream submodule revisions under `build/repos/`
- exports clean source snapshots into `build/sources/`
- builds shared Python and frontend artifacts
- assembles runtime Docker images
- promotes versioned images through testing, staging, and production

That means this repository is the main trust boundary for release provenance. Even though application code lives upstream, this repository decides what gets built, how it gets built, and what gets promoted.

## Why SLSA Matters Here

SLSA is valuable in this repository because the repository already acts like a central release authority, but today it mostly records provenance informally rather than producing verifiable supply-chain evidence.

The main benefits for this repository are:

1. Stronger release traceability.
   For every pushed image, the project could prove which releng revision, which submodule commits, which dependency locks, and which CI workflow produced the artifact.

2. Safer promotion flow.
   Staging and production promotion could move exact immutable digests rather than relying on mutable tags.

3. Better rebuild confidence.
   Reproducibility work already underway in this repository becomes directly useful to provenance, because the same reviewed inputs are more likely to produce the same outputs.

4. Stronger CI trust.
   Provenance emitted by a trusted builder is materially stronger than a human-readable trail of `revision.txt`, `git show`, and container tags.

5. Faster incident response.
   If a dependency, base image, or build tool later turns out to be compromised, release engineering can identify exactly which published digests were affected.

6. Better policy enforcement.
   Vulnerability scan results, SBOMs, image signatures, and provenance can be tied to immutable digests and used as promotion gates.

## Relevant Current State

The repository already has several good foundations.

### Existing strengths

- Submodule revisions are explicit and centrally controlled.
- `build/Makefile` exports clean source snapshots with `git archive` into `build/sources/`.
- Exported source trees include `revision.txt` files, which preserve useful source provenance.
- The top-level `Makefile` records submodule state in `build/submodules.txt` during builds.
- Frontend release builds now rely on committed lockfiles and `npm ci`.
- Backend Python requirements are hash-locked upstream and installed with `--require-hashes`.
- `releng-tool-versions.mk` already pins the `uv` release asset and checksum, showing the repo is willing to treat toolchain inputs as reviewed supply-chain inputs.

### Existing weaknesses

- The active CI workflow explicitly disables BuildKit in `.forgejo/workflows/build-action.yaml`.
- The active CI workflow builds and pushes images, but does not emit structured provenance, SBOMs, or signatures.
- Promotion in `Makefile` is tag-based, not digest-based.
- Container inputs are still partly mutable because several image paths use floating base images and mutable apt resolution.
- The repo has good provenance breadcrumbs, but they are mostly file-based and ad hoc rather than standardized attestations.
- The build pipeline does not yet produce a single release manifest containing image digests, source revisions, lockfile identities, validation results, and attestation references.

## Practical SLSA Position

This repository should not currently be treated as meeting a meaningful SLSA build level.

It does preserve useful provenance information, but it lacks the main properties that make SLSA materially stronger than conventional CI:

- provenance generated by the build platform itself
- immutable artifact identity carried through promotion
- stronger guarantees around builder trust and tamper resistance
- standardized evidence tied to exact outputs

The practical target should not be to claim a formal SLSA level immediately. The useful path is:

1. produce provenance and SBOMs for exact image digests
2. sign images and attestations
3. promote by digest rather than by tag
4. continue tightening reproducibility until provenance describes stable, reviewable inputs rather than partially mutable ones

That sequence delivers most of the operational value before any formal compliance discussion.

## Repo-Specific Gap Assessment

### 1. Builder and CI provenance gap

Current state:

- The active Forgejo workflow checks out the repository, logs into the registry, and runs `make dockers dockers_tagpush REGISTRY=platform.sunet.se`.
- The same workflow sets `DOCKER_BUILDKIT: "0"`.

Gap:

- This blocks straightforward generation of modern container provenance and SBOM outputs.
- The pipeline does not currently produce signed or platform-generated build attestations.

Why it matters:

- The project cannot prove, in a machine-verifiable way, how a digest was built.
- Reviewers and operators are left with logs and conventions rather than first-class supply-chain evidence.

Main files:

- `.forgejo/workflows/build-action.yaml`
- `Makefile`

### 2. Artifact identity and promotion gap

Current state:

- The release flow is testing to staging to production.
- Promotion targets in `Makefile` retag or copy tags between environments.

Gap:

- Promotion is modeled operationally, but not yet around exact immutable digest identity.
- Signatures, SBOMs, and provenance cannot become strong gates if promotions are centered on mutable tags.

Why it matters:

- It is possible to preserve a release process while still losing a precise artifact trail.
- Digest-based promotion is the natural control point for SLSA-style verification.

Main files:

- `Makefile`

### 3. Provenance structure gap

Current state:

- The repository already captures useful provenance hints, including `revision.txt` files and `build/submodules.txt`.

Gap:

- Those records are useful for humans, but they are not emitted as a standardized provenance statement tied to published image digests.
- There is no single release evidence bundle or manifest.

Why it matters:

- Good raw data already exists, but it is not shaped into verifiable release evidence.
- This makes audits, incident response, and policy enforcement harder than necessary.

Main files:

- `build/Makefile`
- `Makefile`

### 4. Reproducibility gap

Current state:

- The repository has improved dependency control for frontend and Python builds.
- The reproducibility note already documents remaining drift sources.

Gap:

- Base images and apt package inputs are still not fully pinned.
- Some build behavior still depends on mutable toolchain or package-resolution inputs.

Why it matters:

- Provenance is most valuable when it describes a build that is stable and reviewable.
- If the same source revisions can still yield materially different results because of drifting base inputs, provenance is informative but weaker as a control.

Main files:

- `docs/BUILD_REPRODUCIBILITY.md`
- runtime Dockerfiles and build Dockerfiles across the repo

### 5. Signing and policy gap

Current state:

- The build path pushes images, but the repo does not currently sign images or attestations.
- There is no policy gate that requires valid signatures, provenance, SBOMs, or scan results before promotion.

Gap:

- The project cannot cryptographically verify that the promoted artifact is the approved artifact.
- Promotion decisions are not yet coupled to supply-chain evidence.

Why it matters:

- SLSA-style provenance becomes much more valuable when it is paired with artifact signing and enforcement.

Main files:

- `.forgejo/workflows/build-action.yaml`
- `Makefile`

### 6. Credential and builder trust gap

Current state:

- Registry login uses a static secret in CI.
- Build trust depends primarily on the CI environment and workflow behavior.

Gap:

- The repository does not yet appear to use short-lived identity-based signing or keyless attestation flows.
- Builder trust is not yet expressed in a way that downstream verifiers can validate automatically.

Why it matters:

- Provenance is strongest when verifiers can trust who generated it and under what workflow identity.

Main files:

- `.forgejo/workflows/build-action.yaml`

## Recommended Adoption Model

The repo should adopt SLSA pragmatically, in phases.

The right first milestone is not formal certification. The right first milestone is a release flow where every published image digest has:

- a machine-readable provenance statement
- an SBOM
- a signature
- a place in a release manifest
- a promotion path that preserves digest identity across environments

## Concrete Implementation Plan

### Phase 1: Prepare the build path for provenance

Goal:

- make the existing CI build path capable of emitting modern metadata

Changes:

1. Re-enable BuildKit or buildx in CI.
   Replace the current `DOCKER_BUILDKIT=0` path with a supported BuildKit-enabled build flow.

2. Capture image digests as structured outputs.
   The build stage should record the exact digest of every pushed runtime image.

3. Define a release manifest.
   Create one machine-readable manifest per candidate build containing:
   - releng commit
   - submodule revisions
   - image names and digests
   - version string
   - toolchain pins from `releng-tool-versions.mk`
   - lockfile identities or hashes where relevant

4. Preserve the existing human-readable provenance breadcrumbs.
   Keep `revision.txt` and `build/submodules.txt`, but treat them as supporting evidence rather than the primary provenance mechanism.

Expected result:

- builds become digest-aware and ready for attestation

### Phase 2: Emit supply-chain evidence

Goal:

- make each produced digest verifiable

Changes:

1. Generate provenance attestations for pushed image digests.
   The attestation should describe the repo revision, workflow identity, builder context, and the resolved build inputs.

2. Generate SBOMs for each runtime image.
   Attach or publish them in a way that keeps them tied to exact digests.

3. Sign the images and attestations.
   Prefer a modern signing workflow that can be verified without ad hoc key distribution where the platform allows it.

4. Publish evidence references alongside build outputs.
   The release manifest should include digest, SBOM reference, signature reference, and provenance reference.

Expected result:

- the repo moves from traceability by convention to verification by artifact evidence

### Phase 3: Move promotion to immutable digests

Goal:

- ensure staging and production receive the exact reviewed candidate artifacts

Changes:

1. Promote by digest instead of by tag.
   Environment tags may still exist for operator convenience, but the promotion decision should be based on immutable digests.

2. Carry evidence forward across environments.
   Promotion should keep the same digest, signatures, SBOMs, and provenance references.

3. Gate promotion on policy checks.
   Require at least:
   - signed image
   - present provenance
   - present SBOM
   - vulnerability policy outcome tied to the digest

4. Record approvals.
   Staging and production promotion should leave a clear approval trail tied to the promoted digests.

Expected result:

- testing, staging, and production become artifact states for the same digest rather than rebuild opportunities

### Phase 4: Tighten reproducibility so provenance has stronger meaning

Goal:

- reduce mutable build inputs that weaken provenance value

Changes:

1. Pin base images by digest.
   Replace floating tags in runtime and build Dockerfiles.

2. Replace floating apt resolution.
   Remove or constrain `dist-upgrade` style behavior and use a reviewed package input strategy.

3. Keep dependency installation fully locked.
   Continue enforcing committed frontend lockfiles and locked Python requirements.

4. Consolidate divergent build paths.
   Bring special-case image paths under the same dependency and toolchain contract where practical.

5. Add focused reproducibility checks.
   Rebuild a narrow slice twice from the same inputs and compare relevant outputs.

Expected result:

- provenance increasingly describes a build that is repeatable, reviewable, and resistant to hidden drift

### Phase 5: Strengthen builder trust and operational enforcement

Goal:

- make provenance and signing usable as enforceable controls rather than just generated files

Changes:

1. Reduce long-lived credential dependence where the platform allows it.
   Prefer shorter-lived, narrowly scoped identity mechanisms for signing and promotion.

2. Define verification points.
   Document where artifact verification happens before promotion or deployment.

3. Fail closed on missing evidence.
   Promotion should stop if required signatures, SBOMs, provenance, or scan results are absent.

4. Document the release evidence contract.
   The repo should describe exactly which evidence is required for a candidate release to advance.

Expected result:

- supply-chain evidence becomes part of release control, not just release documentation

## Recommended Order of Work

The lowest-risk, highest-leverage order is:

1. Enable BuildKit or buildx in CI.
2. Capture exact image digests as structured outputs.
3. Introduce a release manifest.
4. Generate provenance and SBOMs.
5. Sign images and attestations.
6. Change promotion to digest-based operations.
7. Add policy gates for signatures, provenance, SBOMs, and vulnerability results.
8. Continue the reproducibility hardening already identified elsewhere in this repo.

## What This Would Change Operationally

After these changes, a release would no longer be described mainly as:

- a releng checkout
- a set of submodule pointers
- a version tag
- a sequence of environment tags

Instead, a release would be described as:

- a releng commit
- a resolved set of submodule revisions
- a manifest of runtime image digests
- per-digest SBOMs
- per-digest provenance attestations
- per-digest signatures
- policy and approval records bound to those same digests

That is the main practical SLSA benefit for this repository.

## Mapping to Existing Repo Work

This SLSA adoption path builds directly on work already identified in this repository rather than replacing it.

- The build reproducibility effort is a direct prerequisite for stronger provenance value.
- The supply-chain and hardening recommendations already recorded in docs-notes remain valid.
- The existing provenance breadcrumbs in exported source trees are still worth keeping.
- The release promotion model already exists and only needs to move from tag-centric to digest-centric handling.

## Immediate Next Steps

If this repository wants the fastest meaningful SLSA improvement, the next concrete engineering tasks should be:

1. Replace the current BuildKit-disabled CI path with a BuildKit-enabled path that records pushed digests.
2. Add a release manifest artifact to the build job.
3. Prototype provenance and SBOM generation for one runtime image, then expand to all images.
4. Prototype digest-based promotion in parallel with the existing tag-based flow.
5. Make evidence presence a staging and production gate once the output format stabilizes.

## Constraints and Cautions

- Formal SLSA claims should be made only after verifying the exact capabilities of the Forgejo runner, builder isolation model, provenance format, and signing flow.
- The repo can still gain substantial value from SLSA-style practices even before making any formal level claim.
- Provenance is only as strong as the stability of the inputs it describes, so reproducibility hardening should continue in parallel.

## Summary

This repository can benefit significantly from SLSA because it is already the system that decides what eduID release artifacts exist.

Today it has useful provenance hints and improving reproducibility, but it still lacks the core features that make modern supply-chain controls verifiable: builder-generated provenance, artifact signing, digest-based promotion, and policy-gated release evidence.

The recommended approach is to treat SLSA as a phased modernization of the release boundary:

- first enable modern build metadata
- then produce provenance, SBOMs, and signatures
- then promote exact digests
- then keep tightening reproducibility and enforcement

That path fits this repository's architecture and delivers clear operational value even before any formal compliance target is pursued.
